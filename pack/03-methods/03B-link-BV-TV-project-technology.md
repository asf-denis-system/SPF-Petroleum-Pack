# 03B. Связь БВ → ТВ → Проект → Технология

## 1. Фреймворк: четырёхслойная трансформация

Любая инновация в upstream переходит через эти 4 слоя:

```
БВ (Business Вызов)
  ↓ диагностика + выбор метода
ТВ (Technical Вызов) + Набор методов
  ↓ проектирование + команда + датели
Проект (Project) + KPI + Deadline
  ↓ реализация + валидация
Технология/Развёртывание (Deployment Package)
```

**Ключевая идея:** БВ не = ТВ. БВ — это "больно", ТВ — это "лечим техническим путём". Проект — это "как быстро и во что".

---

## 2. Слой 1: БВ (Business Вызов)

### 2.1 Определение БВ
**БВ** — отклонение от плана, выраженное в тоннах нефти, которые **потеряны** или **недополучены**.

### 2.2 Формула БВ
$$\text{БВ}_{\text{тонны}} = \text{План}_{\text{дневной}} \times \text{недопроизв.}_{\%} \times \text{дни}$$

**Пример:**
- План дневной добычи: 500 т/день
- Фактическая добыча: 420 т/день (недопроизводство 16%)
- Период: 30 дней (квартальная отлежка)
- **БВ = 500 × 0.16 × 30 = 2400 тонн потерянной нефти**

### 2.3 Компоненты БВ

| Код БВ | Описание | Геология | Технология | Скважины | ППСО |
|---|---|---|---|---|---|
| **БВ-L** | Локализация: не знаем, где бурить (OOIP ошибка >40%) | ✓✓ | ✗ | ↓ | ✗ |
| **БВ-P** | Проницаемость: пласт закупорен (k → 0.1×k_initial) | ✓ | ✓ | ✓ | ✗ |
| **БВ-D** | Дебит: скважина "встала" или неоптимально (q↓) | ✗ | ✓ | ✓✓ | ✓ |
| **БВ-W** | Вода: обводнённость растёт (WCUT ↑), нужна доработка | ✗ | ✓ | ✓ | ✓ |
| **БВ-E** | Энергия: давление упало, нет закачки воды | ✓ | ✓ | ✗ | ✓✓ |
| **БВ-I** | Инфраструктура: трубопровод забит, нет пропуска | ✗ | ✗ | ✗ | ✓✓ |

### 2.4 Диагностика БВ
**Процесс:**
1. Заметили отклонение (план vs факт)
2. Спросили "почему?" → гипотезы
3. Проверили данные (SCADA, модель, протоколы)
4. Согласовали БВ-код и размер

**Пример диагностики:**
```
Факт: Охват ППД = 40% (план был 60%)
Гипотезы:
  - Давление упало (ППД слабо работает)?
  - Приёмки не слушают команды (управленческое)?
  - Модель неточная (давление OK, охват считали неправильно)?

Проверка: SCADA показывает давление = 32 МПа (норма 33+)
         Объём закачки воды = 800 м³/д (план 900 м³/д)
         Гидродинамическая модель показывает: при 800 м³/д охват ~42%
         
Вывод: БВ-E (энергия упала из-за снижения закачки воды на 100 м³/д)
Размер БВ: 500 т/день × 20% недопроизв. × 30 дней = 3000 т потерь
```

## 3. Слой 2: ТВ (Technical Вызов) + Методы

### 3.1 От БВ к ТВ
**ТВ** — технический параметр, который нужно изменить, чтобы проблема ушла.

| БВ | Симптом | ТВ | Параметр для изменения |
|---|---|---|---|
| **БВ-L** | Не знаем, где нефть (OOIP ошибка) | ТВ-L: Уточнить геометрию | Бурить скважины в новые зоны, обновить ГДМ |
| **БВ-P** | Пласт закупорен асфальтенами/солью | ТВ-P: Вскрыть пласт | Гидравлический разрыв пласта (ГРП) или кислотная обработка |
| **БВ-D** | Скважина не работает (забита, ЭЦН сломана) | ТВ-D: Восстановить поток | Капремонт, газлифт вместо ЭЦН, оптимизация режима |
| **БВ-W** | Вода прорвалась (обводнённость 70%+) | ТВ-W: Отрезать воду | Цементирование, фильтр-пакет, переход на другую перфорацию |
| **БВ-E** | Давление упало | ТВ-E: Увеличить закачку | Нарастить ППД, скважины нагнетания, газлифт |
| **БВ-I** | Трубопровод забит парафином | ТВ-I: Очистить/расширить | Гарячее утепление, дополнительная ёмкость, ингибиторы парафина |

### 3.2 Выбор метода (из 03-methods/)
**Для каждого ТВ выбираем методы из пака:**

```
ТВ-P (вскрыть пласт) → используем:
  UP.M.003: Interference matrix (кто мешает кому из скважин?)
  UP.M.005: Effect monitoring (поставить скважину-монитор, мерить давление)
  UP.M.008: Pilot design (сначала на одной скважине, потом масштаб)
  
ТВ-D (восстановить скважину) → используем:
  UP.M.001: Heterogeneity analysis (есть ли в пласте зоны "хорошие"?)
  UP.M.004: Energy balance (хватит ли энергии для подъёма?)
  UP.M.007: Tech vs state (ЭЦН лучше газлифта в этом состоянии?)
```

### 3.3 ТВ-карта (шаблон)
**Заполняется при постановке проекта:**
```
ТВ-ID: ТВ-E-2024-Q1
Связь с БВ: БВ-E (давление упало на месторождении X)
Метод: ТВ-E.1 — Нарастить ППД с 800 → 950 м³/д
Способ: Доурочить нагнетательную скважину (ТН-25)

Требуемые изменения:
  - Физика: Увеличить расход закачки на 150 м³/д
  - Оборудование: Купить новый насос (мощность 180 кВт)
  - Контроль: SCADA — датчик давления на ТН-25

Связь с технологией (готовый пакет):
  - Насос марка XYZ (производитель, цена, срок доставки)
  - Кабель питания 150 м (тип, медь/алюминий)
  - Монтажные работы (человеко-дни)
  - Обучение персонала (часы)

Риск: Давление может не вырасти, если есть утечки в трубах
Точка контроля (GO–NO-GO): День 10, замер давления — должно быть ≥31.5 МПа
```

## 4. Слой 3: Проект (Project)

### 4.1 Паспорт проекта (Project Charter)
**Фиксирует связь БВ → ТВ → KPI → Deadline:**

```
Проект: "Восстановление охвата ППД месторождения X, Q2–Q3 2024"

БВ: Охват ППД = 40% (план 60%) → потеря 3000 т/30 дней
ТВ: Нарастить закачку воды с 800 → 950 м³/д (увели.откачку на 150)
KPI (критерий успеха): Охват ППД ≥ 58% к концу Q3 (GO-GO) или ≥ 50% (GO-ish)
Deadline: 30 июня 2024 (день G-0, когда новая ТН-25 в работе)

Бюджет:
  - CAPEX: Насос (5 млн ₽) + кабель (200 т ₽) + монтаж (800 т ₽) = ~6 млн ₽
  - OPEX: Электроэнергия (дополнительная) = 150 т ₽/месяц
  
Экономика:
  - Эффект: +15% охвата × семьи скважин = +3000 т/месяц
  - Revenue: 3000 т × 4000 ₽/т = 12 млн ₽/месяц
  - Срок окупаемости: 6 млн ₽ CAPEX / (12 млн ₽ - зарплата и прочее) = ~3–4 месяца
  
Команда:
  - Технолог (руководитель проекта) — планы, мониторинг
  - Скважинник (доурочка) — буровые работы
  - Энергетик (ППД специалист) — контроль давления
  - Инженер ППСО (насос) — монтаж, испытания

Етапы:
  1. Неделя 1–2: Заказ и поставка насоса (внешний поставщик)
  2. Неделя 3–4: Монтажные работы (подсоединение, трубопроводы)
  3. Неделя 5: Испытания, тест-пуск (проверка давления)
  4. Неделя 6–8: Рабочие режимы, наблюдение (мониторинг охвата)
  5. Неделя 9: GO–NO-GO решение на основе SCADA данных

Риск-план:
  - Опасность: Поставка задержится → Альтернатива: ускоренный заказ (дороже)
  - Опасность: Монтаж упрётся во времени → Альтернатива: нанять подрядчика (бюджет +500 т ₽)
  - Опасность: Давление вырастет, но охват не вырастет → Альтернатива: уточнить ГДМ модель

Владелец: Главный технолог месторождения X
Спонсор: Вице-президент по разработке (согласац бюджет)
```

### 4.2 Связь проекта с методами
**Каждый проект обязательно называет, какие методы из пака используются:**

```
Проект: Восстановление охвата ППД

Апликация методов:
  - UP.M.004 Energy balance: Расчёт, хватит ли энергии при 950 м³/д закачки
  - UP.M.005 Effect monitoring: Установка датчика давления (манометр SCADA)
  - UP.M.006 Adaptive management: Если охват <50% на день 15, меняем подход (снимаем скважину ТН-26)
  
Отчет post-project:
  - Охват вырос с 40% → 57% (не дотянули до 58%, но близко)
  - Закачка стабилизирована на 930 м³/д (план 950, но хватает)
  - Осложнения: День 12 сломался датчик давления, заменили
  - Экономика: Вложили 6 млн, получили 12 млн × 4 месяца = 48—6 млн ₽ отбор за сезон ✓
  
Lessons learned:
  - Датчики нужны с резервом (купили 2 запасных)
  - ГДМ модель недооценила эффект ППД на охват (нужно обновить коэфф.)
  - ТН-25 должна была быть доурочена раньше (планирование на 2 месяца, а не 1)
```

## 5. Слой 4: Технология / Deployment Package

### 5.1 Определение
**Deployment Package** — "готовый ящик", содержит всё, что нужно для развёртывания на месторождении.

**НЕ IS:**
- ❌ Идея ("давайте подумаем, может ПОСЗалось")
- ❌ Прототип ("работает в лабе, но на месторождении неизвестно")
- ❌ Half-baked ("купили насос, но не знаем, какой кабель")

**IS:**
- ✅ Спецификация (модель, размер, питание)
- ✅ Поставщик (кто поставляет, цена, гарантия)
- ✅ Монтажные инструкции (чертежи, последовательность)
- ✅ Процедура испытаний (как проверить перед вводом)
- ✅ Руководство оператора (как включить/выключить, что делать при сбое)
- ✅ Техническая подержка (телефон поставщика, гарантия, запасные части)

```yaml
## Компонент 4: Испытания

Процедура тест-пуска (в день нулевой эксплуатации):
  1. Отключить нагнетание (ТН-25 выключена, но давление может быть)
  2. Включить новый насос на 5 минут на холостом ходу (слышать вибрацию, не должно дымить)
  3. Постепенно открывать задвижку (увелич давление с 10 → 20 → 30 → 32 МПа)
  4. Читать показания манометра, записать в журнал
  5. Проверить вибрацию и температуру корпуса (допуск: <60°С)
  6. Если всё OK → режим рабочий на 24 часа
  7. После 24 часов — повторная проверка (вибрация, утечки, давление стабильно?)

Критерий GO:
  ✓ Давление стабильно 31.5–32.5 МПа
  ✓ Расход согласуется с планом 950 м³/д (± 5%)
  ✓ Вибрация низкая (амплитуда < 10 мм/сек)
  ✓ Нет утечек масла/воды
  ✓ Температура корпуса < 60°С

Критерий NO-GO:
  ✗ Давление неконтролируемо скачет (пульсирует ± 2 МПа)
  ✗ Расход меньше 900 м³/д (нужно проверить, что случилось)
  ✗ Вибрация > 15 мм/сек (может быть дисбаланс вала)
  ✗ Утечки масла из корпуса
  ✗ Насос гудит или скрежещет (может попал песок)

## Компонент 5: Руководство оператора

Ежемесячный checklist:
  □ Проверить давление SCADA (норма 31.5–32.5 МПа)
  □ Проверить температуру (норма <55°С)
  □ Проверить расход на счётчике (норма 900–950 м³/д)
  □ Визуально осмотреть насос (нет ли утечек, вибрации нормальной?)
  □ Записать в журнал обслуживания

При сбое (давление упало, расход упал, шум):
  1. Выключить насос (рубильник)
  2. Позвонить в техподержку PUMPS-RU: +7-495-111-2222

  4. Заказать выезд техника (ожидание 2–5 дней в зависимости от удалённости)
```

### 5.3 Масштабирование (от одного места на 10 месторождений)

Когда одного месторождения становится недостаточно:

1. **Стандартизация** — Deployment Package переходит в "типовую" форму (шаблон)
2. **Закупки** — PUMPS-RU может поставлять партию (10 насосов), цена на 10–15% ниже
3. **Обучение** — Один эксперт обучает 10 операторов на 10 местах
4. **Мониторинг** — Центра SCADA (в офисе) следит за 10 насосами одновременно

**Типовой Deployment Package для масштаба:**
```
Спецификация остаётся той же (модель насоса, кабель, монтаж)
Но добавляются:
  - Договор на поставку 10 единиц (скидка 12%, цена 46.8 млн руб)
  - Общий подрядчик (ООО РосМонтаж поедет на все 10 месторождений последовательно, 10 недель)
  - Центральная техподдержка (линия PUMPS-RU 24/7 для всех 10 объектов)
  - Сводный контроль давления из офиса (SCADA агрегирует 10 насосов в одну таблицу)
```

## 6. Полный цикл: БВ → ТВ → Проект → Развёртывание → Метрика

```
ЭТАП 1: БВ ДИАГНОСТИКА (неделя 1)
  Факт vs план → Охват ППД 40% vs 60%
  Анализ → SCADA, ГДМ модель, протоколы
  Вывод → БВ-E: давление упало, закачка недостаточна
  Размер → 3000 т потерь за 30 дней
  Владелец → Главный технолог

ЭТАП 2: ТВ ВЫБОР (неделя 2)
  Метод → Нарастить закачку ППД (ТВ-E.1)
  Методы пака → UP.M.004 (energy balance), UP.M.005 (effect monitoring)
```

## 7. Ошибки при связи БВ → ТВ → Проект (anti-patterns)

### ❌ Ошибка 1: "БВ не диагностирована, сразу в проект"
```
"Охват ППД низкий, давайте купим насос"  ← БЕЗ анализа почему

Последствие: Куплен насос, монтаж, включили, давление выросло на 0.5 МПа, охват вырос с 40% на 42%, проект вроде "работает" но не решил проблему.

Правильно: 
  Потратить неделю на SCADA анализ: давление ОК→40%, проблема не в насосе, а в приёмках (давление на входе ППД ёмкость упало из-за старой трубы). Меняют трубу напорную (дешевле, чем насос).
```

### ❌ Ошибка 2: "ТВ не связана с методами пака"
```
"Хотим увеличить дебит скважины" (ТВ) → "купим новый ЭЦН" (решение) ← БЕЗ анализа, какой метод нужен

Последствие: ЭЦН неправильной мощности, дебит остаётся прежним, деньги потрачены впустую.

Правильно: 
  Применить UP.M.004 (energy balance) — понять, где энергия теряется (трение в трубе? вязкость масла? забито?)
  Применить UP.M.001 (heterogeneity) — может, нефть нужно брать из другой части пласта?
  Только потом выбрать ЭЦН правильной мощности.
```

### ❌ Ошибка 3: "Проект без Deployment Package"
```
"Проект завершён, насос куплен, монтаж закончился" ← но никто не знает, как его включать, к кому звонить при сбое

Последствие: Сломалось оборудование, оператор не знает, что делать, неделю сидит "мёртво" до появления дорогого подрядчика.

Правильно: 
  Deployment Package содержит: спецификация, поставщик, корпус инструкции, телефон техподержки, запасные части.
  Оператор знает: если давление упало → позвонить по номеру XXX, за 48 часов приедет техник.
```

### ❌ Ошибка 4: "Проект БЕЗ GO–NO-GO точки"
```
"Запустили проект, работает уже месяц, охват с 40% стал 45% (план был 60%)"
← когда-нибудь будет 60%? непонятно.

Последствие: Средства "завязаны" в недоделанном проекте, нет решения — продолжить или остановить?

Правильно: 
  Паспорт проекта: "День 15 смотрим SCADA. Если охват ≥50% — GO (проект правился, не снимаем). Если охват <45% → NO-GO (модель неправильна, меняем подход, закрываем проект, анализируем, почему ошиблись)."
```
